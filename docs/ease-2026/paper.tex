%% Abstract

\begin{abstract}
When AI agents fail to build production applications, should we improve the agent or its environment? Recent work suggests that environment quality (templates, tools, guidance) matters more than model selection~\cite{kniazev2025appbuild}. We present a feedback loop for iterative tooling improvement: (1)~an \emph{installable domain knowledge} architecture packages expertise as agent-consumable artifacts; (2)~agents use this package to generate applications, producing execution trajectories; (3)~an \emph{agentic trajectory analyzer} processes these trajectories to identify friction and recommend fixes to the package; (4)~\emph{scripted DevX metrics} serve as the final quality gate, measuring whether generated applications can be run and deployed. We validate on Databricks data applications across two agent backends (Claude Code and Opencode+Sonnet). On 50~data-focused prompts requiring SQL, schema access, and platform integration, we achieve 100\% build and type safety, 98\% runtime success, and 96\% local runability at \$1.61/app---compared to 73.3\% viability on easier consumer prompts in prior work. Extended checks on Opencode apps reveal that 77\% pass runability and 54\% pass deployment validation---apps passing core checks still fail on entry-point discovery and deployment configuration. The trajectory analyzer closes the loop: it processes failure traces, recommends fixes applied via Claude Code, and runability improves from 77\% to 94\% on the expanded set. The architecture targets artifacts amenable to automatic optimization~\cite{gepa2025, khattab2023dspy}, potentially closing the loop without human intervention.
\end{abstract}

\section{Introduction}
\label{sec:intro}

AI agents can generate functional software, but they lack domain-specific knowledge required for production applications. The conventional response is building specialized agents. We take a different approach: improve what agents have access to.

This approach is motivated by recent findings~\cite{kniazev2025appbuild}: when comparing agent performance across environments while holding the model constant, environment quality (templates, tools, guidance) had larger effects than model upgrades. An agent with excellent tooling outperforms a better model with poor tooling.

Accepting that tooling matters raises a question: how do we improve it systematically? Manual inspection doesn't scale. End-state metrics (build pass/fail) don't reveal causes. We need a feedback loop that: (1)~captures how agents actually use tooling; (2)~identifies friction points and root causes; (3)~produces actionable recommendations; (4)~verifies improvements against agent-centric criteria.

We instantiate each component:

\begin{enumerate}
    \item \textbf{Installable Domain Knowledge} (Section~\ref{sec:domain-knowledge}). An architecture for packaging domain expertise as agent-consumable artifacts, exposing tools via CLI commands compatible with emerging skill-based standards.

    \item \textbf{Agentic Trajectory Analyzer} (Section~\ref{sec:trajectory}). A two-phase system: parallel friction extraction with a fast model followed by agentic synthesis with a reasoning model that has source code access.

    \item \textbf{CanItShip: DevX Metrics} (Section~\ref{sec:devx}). The final quality gate: eight shell-scripted metrics (\textbf{Ls1--Ls8}) covering build correctness, runtime health, framework-adaptive runability, and deployment validation.
\end{enumerate}

\textbf{Validation.} We validate across Claude Code (primary, $n{=}47$ of 50 prompts) and Opencode+Sonnet ($n{=}35$ of 50) on Databricks data applications; gaps are due to generation failures and non-standard project structures (Table~\ref{tab:coverage}). On 47~Claude Code applications: 100\% build and type safety, 98\% runtime success, and 96\% local runability at \$1.61/app average. The trajectory analyzer identified improvements that we implemented and verified through multiple iterations.

%% 2. INSTALLABLE DOMAIN KNOWLEDGE

\section{Installable Domain Knowledge}
\label{sec:domain-knowledge}

Developers already use capable agents (Claude Code, Opencode~\cite{opencode2025}). Rather than asking them to adopt yet another tool, we bring domain knowledge to the agents they already use. A user installs our package into their existing environment; the agent gains domain expertise without workflow changes.

The package has three components: context layers that inject domain knowledge progressively, tools exposed via CLI, and a state machine that enforces validation before deployment.

\textbf{Context layers.} Agents have limited context windows. Dumping all domain knowledge upfront wastes tokens and can confuse the model. Instead, we inject context progressively: tool definitions are always available at the protocol level, workflow patterns activate on first discovery call, target-specific constraints load when the application type is detected, and SDK-specific templates inject after scaffolding.

\textbf{CLI tools and MCP parity.} We expose domain functionality through CLI commands, a pattern that Cloudflare~\cite{cloudflare2024codemode} and Anthropic~\cite{anthropic2024tools} found effective. Each CLI command maps 1:1 to a Model Context Protocol (MCP) tool definition, ensuring the same domain knowledge is accessible to any MCP-compliant agent. Lifecycle commands (\texttt{scaffold}, \texttt{validate}, \texttt{deploy}) and data exploration commands (schema discovery, batch queries) are available as both CLI tools and MCP skills, providing the domain-specific capabilities on top of standard workspace tools.

\textbf{State machine.} Applications cannot deploy unless they pass validation after their most recent modification: \texttt{Scaffolded $\to$ Modified $\to$ Validated $\to$ Deployed}. Any change after validation requires re-validation, preventing untested code deployment, a common failure mode when agents skip validation.

\textbf{Agent compatibility.} To validate agent-agnosticism, we tested the package across Claude Code (automated, primary) and Opencode with Sonnet. The key requirement is function calling capability; any agent that can invoke CLI tools works with our package.

%% 3. AGENTIC TRAJECTORY ANALYZER

\section{Agentic Trajectory Analyzer}
\label{sec:trajectory}

The analyzer consumes execution trajectories and recommends package improvements. This closes the error analysis loop~\cite{kravchenko2025mlsystem}: agents struggle $\to$ we see it in trajectories $\to$ we fix the tooling $\to$ agents struggle less.

End-state metrics (build success, test pass) don't reveal whether failures stem from model limitations, tool problems, missing domain knowledge, or underspecified prompts. Trajectories (the sequence of reasoning, tool calls, and results) show where things went wrong. An agent retrying the same malformed SQL five times reveals a missing example. An agent calling N tools for N tables reveals a missing batch operation.

\textbf{Two-phase architecture.} We employ a map-reduce approach optimized for cost and quality. In the \emph{map phase}, each trajectory is processed independently by a fast model (Claude Haiku, ${\sim}\,$\$0.001 per trajectory), extracting errors, retries, confusion patterns, and inefficient tool usage. This runs in parallel. In the \emph{agentic synthesis phase}, aggregated patterns go to a reasoning model (Claude Opus) with read-only access to template source code, tool definitions, and prior analysis history, generating prioritized recommendations.

\textbf{Cost model.} For $N$ trajectories: map costs $N \times {\sim}\$0.001$ (fast model), synthesis costs ${\sim}\$0.5$--3 (reasoning model, bounded at 50~turns). For 50~apps, total analysis cost was under \$15.

\textbf{Human judgment on small N.} On small datasets, we found that the analyzer can overreact, encountering an issue once or twice and recommending building a helper utility without assessing maintenance cost. We recommend human review of recommendations when $N < 50$, treating the analyzer output as a prioritized error analysis rather than an implementation directive.

\section{DevX Metrics}
\label{sec:devx}

DevX evaluation is the final quality gate. After the package is improved and agents generate applications, we need to verify: can these applications be run and deployed?

\textbf{The CanItShip framework.} All eight metrics (Table~\ref{tab:prompts}) produce binary pass/fail verdicts from deterministic commands, spanning three tiers: Core Functionality (Ls1--Ls4), Platform Integration (Ls5), and DevX (Ls6--Ls8). Ls1--Ls6 target the primary agent's stack (Node.js/TypeScript for Claude Code). Ls7--Ls8 extend coverage to Python/Streamlit (Opencode): Ls7 detects the framework and entry point before running a 3-trial install-start-healthcheck protocol, while Ls8 runs \texttt{databricks bundle validate}. Non-deterministic metrics (Ls6, Ls7) pass if at least two of three trials succeed. All metrics are shell-scripted, ensuring reproducibility (see Discussion for our experience with agentic alternatives).

\section{Related Work}
\label{sec:related}

\textbf{Environment over model.} Recent work~\cite{kniazev2025appbuild} found that environment quality had larger effects than model upgrades when comparing agent performance. SWE-agent~\cite{yang2024swe} similarly demonstrated that agent-computer interfaces significantly affect performance. This motivates our focus on improving tooling rather than agents.

\textbf{Evaluation gap.} Existing benchmarks evaluate code correctness (HumanEval~\cite{chen2021evaluating}, SWE-bench~\cite{jimenez2024swe}), task completion (WebArena~\cite{zhou2024webarena}, GAIA~\cite{mialon2023gaia}), or SQL quality (BIRD~\cite{bird2023}, Spider~\cite{yu2018spider}). None ask whether generated code can be operated by other agents, a critical question for compound AI systems.

\textbf{Trajectory analysis.} Analyzing agent execution traces for debugging is established practice (SWE-agent~\cite{yang2024swe}, OpenHands~\cite{wang2024openhands}). Our contribution is making the synthesis phase itself agentic: an agent that explores source code to generate recommendations, rather than static aggregation.

\textbf{Agent evaluation harnesses.} Inspect AI~\cite{inspectai2024} and DeepEval~\cite{deepeval2024} provide evaluation infrastructure but focus on correctness rather than deployment readiness. AgentBench~\cite{agentbench2023} spans multiple environments but does not evaluate autonomous deployability.

\section{Discussion}
\label{sec:discussion}

\textbf{Agent-agnostic vs agent-specific.} Our approach invests in tooling rather than agents: agents improve rapidly (model upgrades are free to adopters), while domain knowledge is slow to encode and hard to transfer. The tradeoff is control; agent-specific approaches can fine-tune behavior precisely, while we rely on the agent's ability to follow tool descriptions. In practice, this suffices for structured tasks but limits open-ended architectural reasoning.

\textbf{Scripted vs.\ agentic evaluation.} We initially prototyped an agentic evaluator (Claude Haiku autonomously running and deploying each app). Agentic evaluation introduced measurement noise---the LLM chose different commands across runs, wrapped status markers in markdown that broke parsers, and produced false positives---conflating evaluator defects with genuine app defects. It was also slow (sequential due to port conflicts) and expensive (token cost per check). However, the prototype was invaluable for \emph{discovering} what to script: it surfaced framework-detection needs and deployment-validation patterns that we encoded as Ls7--Ls8. Prior work~\cite{kniazev2026classification} found that simple, reliable baseline metrics serve as firm ground for reproducible evaluation across large code datasets. Inspired by this, we shifted to a limited set of metrics directly and reproducibly derived from the code---deterministic, parallelizable, and free of LLM cost---reserving agentic evaluation for exploratory settings where evaluation criteria are still being defined.

\textbf{Toward automatic optimization.} Our loop is currently semi-automatic: the analyzer outputs recommendations, a human reviews them. Since recommendations target tool descriptions and prompts, techniques like GEPA~\cite{gepa2025} and DSPy~\cite{khattab2023dspy} could potentially close the loop without human intervention.

\textbf{Limitations.} \emph{External validity:} We validate on Databricks; the harness is platform-independent but domain packages require adaptation. \emph{Internal validity:} Shell metrics cover $n{=}47$ Claude and $n{=}35$ Opencode apps; extended metrics (Ls7--Ls8) cover $n{=}18$ Opencode Set~1. Scaling to 100+ is planned. \emph{Construct validity:} Even deterministic scripts exhibit timing variance; multi-trial protocols are essential. The feedback loop is demonstrated via case studies (Table~\ref{tab:improvements}); longitudinal quantification is planned.

\section{Conclusion}
\label{sec:conclusion}

We presented a feedback loop for iterative improvement of agent tooling: installable domain knowledge, an agentic trajectory analyzer, and CanItShip metrics. Using shell-scripted evaluation (Ls1--Ls8) across 82~Databricks applications and two agent backends (Claude Code $n{=}47$ of 50~prompts, Opencode+Sonnet $n{=}35$ of 50; remainder excluded due to generation failures and non-standard structures), we achieve 100\% build and type safety, 98\% runtime success, and 96\% local runability at \$1.61/app. Extended checks on Opencode apps ($n{=}18$) show 77\% runability and 54\% deployment validation before fixes, with failures traceable to missing entry points and deployment manifests. We demonstrate the closed loop: the trajectory analyzer identifies failure patterns, fixes are applied via Claude Code, and runability improves from 77\% to 94\%. When agents fail, improve their environment rather than the agents themselves. The evaluation harness, trajectory analyzer, 50-prompt benchmark, and anonymized artifacts are available as open source.\footnote{Repository URL withheld for double-anonymous review; will be included in the camera-ready version.}
